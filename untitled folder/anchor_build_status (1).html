WHAT'S DONE (tested and working pre-hackathon)
DONE 1. Window watcher
Reads the active window title on the user's computer every 5 seconds. Just the title, not screen content.
How it works
Uses macOS osascript to ask System Events for the frontmost application name and its window title. Returns a string like "Google Chrome - Reddit". Has a fallback that returns just the app name if the window title isn't accessible. Runs inside the monitoring_loop() as get_active_window_title().
result = subprocess.run(["osascript", "-e", script], capture_output=True, text=True, timeout=5) return result.stdout.strip() # "Google Chrome - Reddit"
TESTED: Correctly reads window titles from Chrome, Terminal, VS Code, WhatsApp, Preview. Accessibility permissions sorted (System Settings > Privacy > Accessibility > Terminal).
File: server.py, function: get_active_window_title()
DONE 2. Task context builder
Analyzes the user's task at session start and returns what domain it involves, what tools/sites are likely needed, and what type of activity it is.
How it works
Sends the task description to Gemini 2.5 Flash with a prompt asking it to return a JSON object with: task (cleaned up), domain, likely_tools, likely_sites, activity_type (reading/writing/coding/browsing/mixed), and always_ok apps. The prompt explicitly says to NEVER include messaging apps in always_ok. This context is stored in session_state["task_context"] and used by the classifier for every window classification during the session.
Input: "filling out volunteer application" Output: { "task": "Filling out volunteer application", "domain": "civic engagement, volunteering", "likely_tools": ["Web Browser", "PDF Reader", "Email"], "activity_type": "mixed", "always_ok": ["Spotify", "Calculator"] }
File: server.py, function: create_task_context()
DONE 3. Content-aware LLM classifier
Classifies each window as relevant, drift, or unsure based on the CONTENT in the title, not just the app name.
How it works
Takes task_context + window_title and sends to Gemini with strict rules: (1) Read the content in the title, not just app name -- "Claude - skills assessment" = relevant, "Claude - yhack idea" = drift for same app. (2) Messaging apps (WhatsApp, Telegram) = "unsure" by default. (3) Social media = "drift" unless title content clearly matches task. (4) Terminal = "relevant" only if task involves coding. (5) AI tools = read the conversation title. Returns JSON with verdict, confidence (0-1), and reason.
TESTED: 15/15 correct classifications including tricky cases. "Claude - personal skills" = relevant, "Claude - yhack idea" = drift. WhatsApp = unsure. Reddit = drift. Volunteer application form = relevant.
File: server.py, function: classify_window()
DONE 4. Classifier caching
Same window title never gets re-classified. Saves money and latency.
How it works
A Python dictionary classification_cache stores results keyed by window title. Before calling Gemini, checks the cache first. If found, returns instantly with from_cache: True. Cache resets at the start of each session (different task = different relevance).
First "Chrome - Reddit": 1,431ms (Gemini call) Second "Chrome - Reddit": 0ms (cache hit)
File: server.py, inside classify_window()
DONE 5. Anchor Agent (ADHD specialist)
Reads full session history + session state and reasons about what to do. Has deep ADHD neuroscience knowledge. Only called on drift/unsure/timeout events.
How it works
The agent prompt gives Gemini an ADHD specialist persona that understands Barkley's executive function model, time blindness, and task initiation paralysis. It receives: Session state (drift count, elapsed time, ever_on_task, expected notifications, recent nudge count). Full observation history (every window change, every nudge sent, every user response -- last 25 events). Latest event (what just happened). The agent diagnoses WHY the user is drifting (depleted, stuck, overwhelmed, bored, avoiding, notification pull) and picks a task-specific response. Returns JSON with action (speak/ask/suggest_break/stay_silent), message, and reasoning.
TESTED LIVE: Agent correctly stayed silent on 1st drift, spoke on 2nd, escalated on 3rd. Fixed 3 bugs: (1) agent never speaking, (2) WhatsApp as relevant, (3) Claude with wrong title as relevant. Also fixed agent hallucinating app names -- now uses ACTUAL app from event.
File: server.py, function: anchor_agent_decide()
DONE 6. ElevenLabs voice
Agent speaks nudges out loud through the user's speakers. Voice works even when the Anchor tab is hidden.
How it works
When the agent decides to speak, the message text is sent to ElevenLabs API which generates audio using the Roger voice (laid-back, casual). The play() function runs in a separate thread via asyncio.to_thread() so the monitoring loop doesn't stall while audio plays. Requires mpv installed for audio playback.
audio = elevenlabs_client.text_to_speech.convert( text=message, voice_id="CwhRBWXzGAHq8TQ4Fs17", # Roger model_id="eleven_multilingual_v2", output_format="mp3_44100_128", ) await asyncio.to_thread(play, audio)
TESTED: Voice plays through speakers. Heard 5 different nudge messages. test_voice.py confirms all messages play correctly. 10,000 free credits available.
File: server.py, function: speak(). Also test_voice.py for standalone test.
DONE 7. Progressive cooldown (anti-nagging)
Cooldown between nudges gets longer each time. Prevents nagging which ADHD research shows makes things worse.
How it works
Counts how many nudges were sent in the last 5 minutes. Based on the count, sets a cooldown period: 0-1 nudges = 30 sec, 2 nudges = 60 sec, 3 nudges = 120 sec, 4+ nudges = 300 sec (5 min). This check runs BEFORE calling the LLM, so during cooldown periods there's zero API cost. The nudge count is also passed to the agent prompt so it knows to switch strategy (suggest break instead of another redirect) when nudges aren't working.
File: server.py, inside anchor_agent_decide()
DONE 8. Drift escalation
Agent escalates responses: 1st drift = silent, 2nd = nudge, 3rd+ = must intervene.
How it works
The drift_count in session_state increments every time the classifier returns "drift". This count is passed to the agent prompt. The prompt rules say: 1st drift = stay silent, 2nd = speak, 3rd+ = must intervene even if user self-corrected. The agent ALSO varies its phrasing every time -- it checks previous messages in the history and uses different words, tone, and approach.
TESTED: Confirmed agent stays silent on 1st drift, speaks on 2nd. Previously had a bug where agent treated every drift as "first" -- fixed with explicit escalation rules.
File: server.py, prompt rules inside anchor_agent_decide()
DONE 9. Sustained drift detection
If user opens a drift app and STAYS there for 1+ minute, agent speaks even if it was the first drift.
How it works
In the else branch of the monitoring loop (same window, no change), checks if the current window was classified as "drift" in the cache. If yes, starts a timer (sustained_drift_start). If the user stays on that drift app for 1+ minute, fires an event to the agent. The sustained_drift_nudged flag prevents re-triggering. Both flags reset when the window changes.
File: server.py, monitoring_loop() else branch
DONE 10. Task initiation detection
If 1+ minute passes and user never opened a task-relevant app, agent helps them start.
How it works
Checks ever_on_task (only becomes True when classifier sees a relevant window) AND task_initiation_nudged (prevents re-trigger) AND elapsed >= 1 minute. If all three conditions met, fires "task initiation paralysis" event to agent. The agent suggests the smallest possible first step specific to the task: "Just open the form and fill in your name."
TESTED: At 1.0 min with user on Terminal, agent correctly said: "Looks like you haven't opened your volunteer application yet. Would you like to open it now?"
File: server.py, monitoring_loop() else branch
DONE 11. Silent drift detection (pynput)
Correct window is open but no keyboard/mouse activity. User picked up phone or zoned out.
How it works
pynput listeners run in background threads, tracking keyboard presses, mouse moves, clicks, and scrolls. Each event updates last_activity_time. The monitoring loop checks how long since last activity. Threshold adapts to task type: writing/browsing/mixed = 1 min, coding = 3 min, reading = 7 min. Fires event to agent. idle_nudged flag prevents re-trigger, resets on window change.
idle_thresholds = { "writing": 1, # constant typing expected "coding": 3, # sometimes you think first "browsing": 1, # should be clicking/scrolling "reading": 7, # legitimately staring at screen "mixed": 1, # default }
File: server.py, pynput listeners at top + monitoring_loop() else branch
DONE 12. Notification pull detection
Detects if a window switch was a notification popping up (WhatsApp call) vs user deliberately navigating.
How it works
Tracks last_window_change_time and compares with last_activity_time. If window changed within 3 seconds of last keyboard/mouse activity on the previous window, it was probably a notification pull. This info is included in the event summary sent to the agent: "NOTIFICATION PULL" or "User deliberately navigated here." The agent prompt says to be extra gentle with notification pulls.
File: server.py, monitoring_loop() if title != last_title branch
DONE 13. Hyperfocus detection
40+ minutes focused without break. The opposite problem from drift -- ADHD hyperfocus burns all executive function.
How it works
Tracks last_break_time in session state (initialized to session start time). Calculates minutes since last break. If >= 40 AND ever_on_task is True, fires event to agent. Updates last_break_time to prevent re-trigger.
File: server.py, monitoring_loop() else branch
DONE 14. Long stay detection on relevant apps
On Claude/ChatGPT/YouTube for 15+ minutes. Classified as relevant but might have drifted WITHIN the app.
How it works
Tracks relevant_window_start and last_relevant_window. If the current relevant window contains "claude", "chatgpt", "youtube", or "openai" and time_on_window >= 15 minutes, fires event to agent. Resets the timer after firing so it doesn't re-trigger every 5 seconds.
File: server.py, monitoring_loop() else branch
DONE 15. Break management
User can take a 5-minute break. Monitoring pauses. Agent reminds when break is over.
How it works
When user clicks "Taking a break" (or says it via voice), break_active = True and break_start is set. The monitoring loop skips all checks during break. Every 5 seconds it checks if break_duration >= 300 seconds. When break ends, speaks "Break's over. Ready to get back?" and resets break state.
File: server.py, monitoring_loop() break check + WebSocket handler
DONE 16. Task-specific agent responses
Agent gives responses specific to the user's actual task, never generic "get back to work."
How it works
The agent prompt includes the user's exact task and has examples of task-specific chunking: forms = "fill the first field", reading = "read the abstract", coding = "write the function signature". The agent diagnoses the drift reason (depleted/stuck/overwhelmed/bored/avoiding) and picks a response specific to both the reason AND the task. Also acknowledges task-specific tedium: "Forms are repetitive" vs "Dense papers are hard to stay with."

DONE 17. Anti-hallucination rules
Agent must use actual app names from events, never makes up apps the user hasn't visited.
How it works
Prompt rules state: "ONLY reference apps that appear in the session history. Use the ACTUAL app name from the LATEST EVENT." Also: "If ever_on_task is False, do NOT say 'you left your task' because they never opened it." Previously the agent would say "you left your application for Reddit" when the user had never opened the application and wasn't on Reddit. Fixed by making these rules explicit.

DONE 18. Session summary builder
Calculates real stats from session data when session ends.
How it works
Iterates through observation_history to calculate: total time, focused time, drift count, nudge count, longest focus streak (tracks streak start/end from relevant/drift transitions), top drift trigger app (counts drift app occurrences with Counter), and builds a segmented timeline (focused/drift/break segments with start/end times). Returns a dict that gets sent to the frontend.
File: server.py, function: build_session_summary()
DONE 19. FastAPI server + WebSocket
REST endpoints for session start/end/status. WebSocket for real-time frontend updates.
How it works
POST /session/start builds task context, initializes session state, starts monitoring loop as asyncio background task. POST /session/end stops monitoring, builds summary. GET /session/status returns current state. WS /ws accepts WebSocket connections from React frontend. The broadcast() function sends events to all connected clients. WebSocket also listens for button clicks (pull_me_back, taking_break, im_ready).
TESTED: All endpoints working via curl. Session starts, monitoring runs, classifications print, agent decides, voice speaks, session ends with summary.
File: server.py, REST endpoints + websocket_endpoint()

How to do it -- 4 touch points
Touch point 1: Start session. When user clicks "Let's begin" on the React UI, call POST /session/start with the task, duration, DND setting, and expected notifications.

Touch point 2: Listen for events. After session starts, open a WebSocket to ws://localhost:8000/ws. Listen for incoming events and update the UI: type: "classification" = update the status dot (green/red). type: "nudge" = show the nudge card with the message. type: "status" = update focused/drifted state. type: "session_ended" = show session summary screen with real stats. type: "break_started" = show break timer.

Touch point 3: Send button clicks. When user clicks a button on a nudge card, send via WebSocket: ws.send(JSON.stringify({action: "pull_me_back"}))

Touch point 4: End session. When user clicks "End Session", call POST /session/end.
// Touch point 1: Start session const res = await fetch("http://localhost:8000/session/start", { method: "POST", headers: {"Content-Type": "application/json"}, body: JSON.stringify({task, duration, dnd, expected_notifications}) }); // Touch point 2: Listen for events const ws = new WebSocket("ws://localhost:8000/ws"); ws.onmessage = (event) => { const data = JSON.parse(event.data); if (data.type === "nudge") showNudgeCard(data.message); if (data.type === "status") updateStatusDot(data.value); if (data.type === "session_ended") showSummary(data.summary); }; // Touch point 3: Send button clicks ws.send(JSON.stringify({action: "pull_me_back"})); // Touch point 4: End session await fetch("http://localhost:8000/session/end", {method: "POST"});
